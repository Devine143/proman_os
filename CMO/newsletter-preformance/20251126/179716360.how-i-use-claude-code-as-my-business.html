Three months later, Claude Code has become my silent business partner.
Not for coding—for everything else.

Why a “coding tool” works for project management

Here’s what makes Claude Code different from normal AI tools:

Regular AI chat:
– Lives in a browser tab
– You upload documents manually
– You implement all changes yourself
– Context resets constantly
– Every conversation starts from zero

Claude Code:
– Lives in your actual file system
– Reads entire project folders automatically
– Writes and edits documents directly
– Remembers your workflow through config files
– Builds on its own previous outputs

If traditional AI is like visiting a consultant, Claude Code is like having a colleague sitting next to you with access to your whole project environment.

For example, I can say:

“Read the tender documents in this folder, analyse the scope, and create a preliminary risk register based on what you find.”

Claude Code actually does it—reading all 47 pages across multiple documents, understanding the project, and producing a draft I can use immediately.

Important caveat: You need minimal comfort with the command line. If you’ve never opened a terminal, there is a learning curve—but the payoff is enormous.

My actual PM workflows with Claude Code

These are the real ways I use Claude Code daily—where it excels, and where it has burned me.

1. Risk assessment with broader coverage

The old way: hours of reading tender docs, site reports, proposals, and manually identifying risks.

With Claude Code:

“Read all documents in /Tender. Identify risks across site conditions, regulatory compliance, stakeholder conflicts, budget, and timeline. Draft a risk register with probability, impact, and mitigations based on my standard.”

Fifteen minutes later I had a clean draft.
It even found a buried line on page 38 of a geotech report about soil contamination testing—something that could have delayed approvals by two months.

Key point:
Claude produces drafts, not decisions.
I validate every risk with professional judgement.

2. Stakeholder analysis with depth

Stakeholder management is everything in construction.
I point Claude Code at my project folder and ask:

“Based on project charters and meeting notes, create a stakeholder matrix with interests, influence, concerns, and engagement strategies. Flag potential conflicts.”

Claude produces a surprisingly contextual analysis.
But here’s the limitation:

Once, it flagged a contractor as “high conflict risk” based on a months-old dispute documented in notes—even though the relationship is now excellent.

Lesson:
Claude understands documents, not human dynamics.
You still own the relational intelligence.

3. Project charters that consolidate real information

Instead of spending a day compiling a charter, I now say:

“Read the kickoff notes, proposal, and site assessment. Draft a project charter following PMBOK: purpose, scope, deliverables, constraints, assumptions, timeline, and success criteria.”

Claude produces a solid baseline.
I refine the strategic parts, validate details, and adjust tone for the client.

Professional responsibility reminder:
AI supports your expertise—it doesn’t replace accountability.

4. Work breakdown structures that start strong

Creating a WBS from scratch is tedious.
Now I say:

“Based on scope-statement.md, create a WBS to level 3/4, including design, approvals, site establishment, structure, envelope, finishes, externals, and handover.”

Claude produces a structured WBS following typical construction sequencing.

Where it failed:
For a heritage renovation, it completely omitted Heritage Western Cape approvals.
It assumed regular new construction.

I now specify: “This is a heritage building with HWC requirements.”

5. Meeting preparation and follow-up

Before meetings:

“Read previous minutes + issues log + status report. Prepare an agenda with decisions needed and talking points.”

After meetings:

“Transform my rough notes into formal minutes with action items and due dates. Update issues log.”

Admin tasks that used to drain hours now take minutes.

The setup that makes this work

Claude Code isn’t magic—you need structure.

Project folder structure

I use a PMBOK-aligned structure:

Project/
│
├── 01-Initiation/
├── 02-Planning/
├── 03-Execution/
├── 04-Monitoring/
└── CLAUDE.md

Everything else hangs off this.

The document skills game-changer

Anthropic’s document skills let Claude create:
Word docs, Excel sheets, PowerPoints, PDFs
—directly from your markdown.

My workflow:
Draft fast in markdown → export polished client documents.

“Convert project-charter.md into a formatted .docx with cover page, TOC, and styled tables.”

Result: client-ready documents.

They’re 80–90% perfect. I do a final polish.

The CLAUDE.md file (your secret weapon)

This file trains Claude on your project:

# Project Context
## Overview  
## Stakeholders  
## Current Phase  
## Standards  
## Preferences

It takes 5 minutes to update but dramatically improves quality.

Claude gives better outputs when it knows you.

Who this is actually for

– Your work is information-heavy
– You want to eliminate admin drag

Not for you if:
